# Primary Data Build Checklist

Scope: primary data only. This means the original data the app must collect, own, validate, sync, or preserve. Derived data such as search indexes, embeddings, recommendations, dashboards, analytics summaries, generated reports, caches, and rebuildable manifests are secondary data and are intentionally excluded here.

## Primary Data Inventory

| Order | Primary data set | What it contains | Source of truth | Time profile | Expected volume | Cost drivers | Required before | Failure risks | Build checklist |
|---:|---|---|---|---|---|---|---|---|---|
| 1 | Canon source texts | Raw nikaya/sutta text, IDs, book/chapter structure, titles, paragraphs, translations if treated as app content | Versioned local `data/` plus cloud object storage | Mostly static; batch updates during corpus releases | High file count, moderate JSON size | Storage, validation time, manual text QA | Browse, sutta pages, quiz linking, search | Missing suttas, duplicate IDs, broken hierarchy, encoding issues | Define canonical ID format; store raw imports; validate schema; keep immutable snapshots; record source/license/version; add tally counts by nikaya/book |
| 2 | Teacher audio files | Original audio assets for suttas, playlists, duration, file names, public object paths | GCS audio bucket or equivalent object storage | Static to slow-changing; large initial upload | Very high storage/bandwidth relative to JSON | Audio storage, egress, transcoding, CDN/cache | Audio player, listen progress, audio-linked practice | Missing files, path mismatch, CORS errors, large mobile downloads | Standardize path convention; verify MIME/CORS; store duration; test mobile streaming; create missing-file report; keep upload manifest |
| 3 | Audio segment mappings | Start/end timestamps linking text segments to audio | Versioned JSON/CSV in pipeline output | Batch-generated; changes when audio/text alignment changes | Medium; one mapping per audio-backed sutta | Manual correction time, validation tooling | Highlighted audio, segment navigation, accurate listen state | Bad timestamps, drift, out-of-order segments, mismatched IDs | Validate monotonic timestamps; check segment coverage; store confidence/manual-review flag; keep source audio version |
| 4 | App content records | Built-in quizzes, practice prompts, teacher-aligned answers, book-one practices, reflection prompts, local images/content pages | Git-tracked `src/data` and assets | Slow-changing editorial releases | Low to medium | Editorial QA, test coverage | Quiz, practice, reflection, onboarding content | Stale content, broken sutta references, answer ambiguity | Give each item stable ID; link to canonical sutta ID; validate references in tests; add author/reviewer/date fields where useful |
| 5 | User auth account | Email auth identity, Supabase user ID, auth provider metadata | Supabase Auth | Created once, updated on login/security events | One row per user | Auth provider, email delivery | Sync, profile, user-owned data | Account duplication, auth callback failure, password reset failure | Configure redirects; test signup/login/reset; document RLS assumptions; handle anonymous/local-only users separately |
| 6 | User profile | Username, display name, avatar URL, created/updated timestamps | `profiles` table | Created during onboarding; occasional edits | One row per user | Database rows, avatar hosting if used | Profile page, personalization, social/future sharing | Username conflicts, incomplete onboarding, invalid avatars | Enforce unique username; validate length/characters; upsert safely; sync local draft to server after login |
| 7 | Reading progress | Per-user/per-sutta open count, opened time, read time, updated time | `reading_progress` table plus local offline buffer | High-frequency small writes; sync debounced | Up to users x opened suttas | Database writes, sync retries | Continue reading, read state, tree/progress UX | Lost offline progress, clock skew, duplicate writes, overwrite with older state | Define merge rule using `updated_at`; debounce writes; queue offline events; test conflict cases; add RLS |
| 8 | Audio listen progress | Per-user/per-sutta listen fraction and updated time | `audio_listen_progress` table plus local offline buffer | High-frequency during playback; should be throttled | Up to users x listened suttas | Database writes if not throttled | Resume audio, audio completion, practice state | Write spam, fraction regression, invalid values | Clamp 0..1; throttle updates; only advance unless explicit seek policy says otherwise; test resume on another device |
| 9 | Quiz attempts and leaf state | Sutta quiz attempt, selected answer, status/color, review due time, fixed/gold state | Local storage now; should become server table for sync | User-generated; medium frequency | Users x quiz attempts/leaves | DB rows, sync complexity | Tree, spaced review, mastery loop | Local-only data loss, duplicate attempts, bad decay timing | Create `quiz_attempts`/`leaf_states`; store attempt time and review state; define decay/review rules; sync after login; write tests for state transitions |
| 10 | Reflection answers | User reflection prompt, answer text, selected sutta/context, created time, optional privacy state | Future server table plus local draft storage | User-generated; low to medium frequency | Potentially large text per active user | Storage, privacy/security review | Reflection history, AI feedback, personal practice log | Sensitive text leakage, draft loss, accidental public exposure | Mark private by default; encrypt or restrict strongly with RLS; autosave drafts; add delete/export; avoid logging full text into analytics |
| 11 | AI harness traces | User input, intent, steps, outputs, feedback, duration, run metadata | `harness_traces` table | Event-based; can grow quickly | Medium to high depending AI usage | Database storage, retention, privacy review | Debugging AI flows, quality improvement | Capturing sensitive text, runaway trace size, unbounded retention | Set retention policy; cap payload size; redact secrets; store model/version; add user delete path; separate production vs dev traces |
| 12 | Device identity | Local device ID used for sync/log attribution | Local storage, optionally server-linked | Created once per browser install | One record per device/browser | Minimal | Offline queue, UX events, debugging sync | Reset on storage clear, collision, cross-user leakage on shared devices | Generate random ID; do not treat as authentication; rotate/reset on logout if needed; never store secrets in it |
| 13 | User settings | Theme, language, playback preferences, reading preferences, dev/fresh-start flags where user-owned | Local storage now; future `user_settings` table | Low-frequency edits | One row/document per user/device | Minimal | Personalized reading/audio UX | Settings lost across devices, invalid enum values | Define schema and defaults; validate on load; migrate versions; decide which settings are per-user vs per-device |
| 14 | UX event log | Raw user interaction events: page views, taps, errors, feature usage, timing | `ux_events` table plus local queue | High-frequency append-only | Potentially very high | DB writes, storage, retention | Product learning, debugging, funnel analysis | Over-collection, write cost, privacy risk, noisy events | Create event taxonomy; sample noisy events; batch uploads; omit sacred/private content; add retention and opt-out |
| 15 | Operational failure records | Sync failures, pipeline failures, missing asset reports, validation errors, deployment smoke results | Logs plus durable `ops_events` or issue artifacts | Burst during deploy/pipeline; low during normal use | Low to medium | Log storage, alerting | Reliable releases, data repair | Failures hidden in transient logs, no owner, repeated regressions | Store structured error type/source; link to build/run ID; alert on critical counts; keep repair checklist |

## Build Plan

| Phase | Goal | Primary data to build | Time estimate | Exit criteria | Tests/validation |
|---:|---|---|---|---|---|
| 1 | Freeze canonical IDs and schema contracts | Canon source texts, app content records | 1-2 days | Every sutta/content item has stable ID, source, version, and schema | JSON schema tests; duplicate ID tests; broken-reference tests |
| 2 | Make corpus and audio deployable | Canon texts, teacher audio, audio mappings | 2-5 days depending audio volume | Validated JSON and audio are uploaded; CORS works; global tally passes | Pipeline tally; random mobile playback check; missing file report |
| 3 | Make user identity reliable | Auth account, profile | 0.5-1 day | Signup, login, onboarding, reset, logout all work | Unit tests for profile helpers; e2e auth smoke |
| 4 | Add durable progress sync | Reading progress, audio listen progress, device identity | 1-3 days | Progress survives reload, logout/login, and second device sync | Merge-rule unit tests; offline queue tests; Supabase RLS tests |
| 5 | Server-sync learning loop | Quiz attempts, leaf state | 2-4 days | Tree state survives device changes and review timing is deterministic | Leaf transition tests; duplicate-attempt tests; migration from localStorage |
| 6 | Protect personal practice data | Reflection answers, settings | 2-4 days | Private reflections/settings save, edit, delete, and sync safely | RLS tests; autosave tests; delete/export tests |
| 7 | Observe without over-collecting | UX events, AI harness traces, failure records | 1-3 days | Events are batched, bounded, useful, and privacy-reviewed | Event schema tests; payload size tests; redaction tests |
| 8 | Release hardening | All primary data sets | 1-2 days per release | Build, tests, pipeline tally, deploy smoke, and rollback notes are complete | `npm run test`; coverage target near 85%; `npm run verify`; smoke auth/audio checks |

## Data Quality Checklist

| Check | Applies to | Pass condition |
|---|---|---|
| Stable IDs | Suttas, audio, quizzes, leaves, reflections | IDs do not change between releases unless a migration exists |
| Ownership | User data tables | Every row has clear owner or explicit public/admin status |
| Sync rule | Progress, leaves, settings, reflections | Conflict rule is documented: latest timestamp, max progress, append-only, or manual merge |
| Time rule | Progress, reviews, traces, events | Store UTC server timestamps; tolerate client clock skew |
| Volume rule | Audio, events, traces | Large/high-frequency data is batched, throttled, sampled, or stored in object storage |
| Cost rule | Audio, events, traces, embeddings later | Estimate storage, egress, and write volume before enabling at scale |
| Failure rule | Pipeline, sync, auth, deploy | Failures are captured as structured records with retry/repair path |
| Privacy rule | Reflection, AI traces, UX events | Sensitive text is private by default, redacted from logs where possible, and deletable |
| Migration rule | Local storage to server | Existing local data is imported once, deduped, then marked synced |
| Test rule | All primary data modules | Critical merge, validation, and permission logic has tests; aim for 85% coverage on data/sync modules |
